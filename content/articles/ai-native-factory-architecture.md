# AI Native 工厂的参考架构（附架构图）

过去十年，制造业经历了从"信息化"到"数字化"的转型：ERP 上线了、MES 部署了、IoT 传感器装了。但大多数工厂的 AI 应用仍然停留在"打补丁"阶段——在现有系统上外挂一个预测模型，在一个看板上加一个异常告警。

**AI Native 工厂不是"工厂 + AI"，而是"以 AI 为内核重新设计的工厂"。** 就像移动互联网时代的 App 不是"网站 + 触摸屏"，而是从交互逻辑到商业模式都被重新定义的产品。

本文将给出一个完整的 AI Native 工厂参考架构。

---

## 一、什么是 AI Native 工厂？

### 定义

AI Native 工厂是指：**在工厂的规划、建设、运营全生命周期中，将 AI 能力作为基础设施层进行系统性设计，使数据采集、模型推理、智能决策像水电一样即取即用。**

### 与传统智能工厂的区别

| 维度 | 传统智能工厂 | AI Native 工厂 |
|------|------------|----------------|
| AI 角色 | 外挂应用 | 基础设施 |
| 数据流向 | 系统 → 人 → 决策 | 系统 → AI → 人（审核） |
| 响应速度 | 分钟/小时级 | 毫秒/秒级 |
| 决策范围 | 单点优化 | 全局协同优化 |
| 模型管理 | 一次性开发 | 持续训练/迭代 |
| 扩展方式 | 项目制 | 平台化 |

### 核心特征

1. **数据即资产**：所有生产活动产生的数据都被结构化采集、治理和资产化
2. **模型即服务**：AI 模型以 API 形式提供，任何业务系统都可调用
3. **边缘协同**：边缘 AI 负责实时推理，云端 AI 负责模型训练和复杂分析
4. **闭环自优**：从数据采集 → 模型推理 → 决策执行 → 效果反馈形成闭环
5. **人机协同**：AI 提供决策建议，人做最终决策和异常处理

## 三、第一层：数据基座

数据基座是 AI Native 工厂的地基。没有高质量的数据基座，上层的 AI 应用就是空中楼阁。

### 3.1 数据采集

**多源异构数据统一接入：**

| 数据源 | 采集方式 | 频率 | 数据特征 |
|--------|---------|------|---------|
| PLC/传感器 | OPC UA / Modbus | 1ms-1s | 时序数值 |
| MES | API / 数据库同步 | 秒-分钟 | 事务型 |
| ERP | API / ETL | 小时-天 | 事务型 |
| 视觉检测 | 工业相机 | 帧/秒 | 图像 |
| 设备日志 | Syslog / File | 事件驱动 | 文本 |
| 环境监测 | IoT 网关 | 分钟 | 数值 |

**关键技术：**
- **边缘网关**：部署在产线侧，负责协议转换、数据清洗、本地缓存
- **时序数据库**：如 InfluxDB、TDengine，专门处理高频时序数据
- **消息队列**：如 Kafka，保证数据不丢失、可回放

### 3.2 数据治理

数据治理解决"数据能用"的问题：

```
原始数据 → 清洗 → 标准化 → 质量校验 → 资产化
            │         │          │           │
            ▼         ▼          ▼           ▼
         去重/去噪  单位统一   完整性检查   元数据管理
         缺失填充   编码规范   准确性校验   数据目录
         异常过滤   命名规范   时效性检查   血缘追踪
```

**核心原则：**
1. **一次采集，多处使用**：建立统一数据湖，避免"烟囱式"数据管道
2. **元数据驱动**：每个数据字段都有清晰的业务定义、采集方式、质量规则
3. **数据质量可度量**：建立数据质量评分卡，量化数据可用性

### 3.3 数据存储

**分层存储策略：**

```
热数据（最近7天）   → 时序数据库（InfluxDB/TDengine）  → 毫秒级查询
温数据（7-90天）    → 列式数据库（ClickHouse）          → 秒级查询
冷数据（90天以上）  → 对象存储（MinIO/S3）+ Parquet     → 分钟级查询
```

## 四、第二层：AI 平台

AI 平台层是 AI Native 工厂的"发动机"，负责模型的全生命周期管理。

### 4.1 模型管理（MLOps）

```
模型生命周期：
  数据准备 → 特征工程 → 模型训练 → 模型评估 → 模型部署 → 在线推理 → 效果监控 → 重训练
     ↑                                                                                    │
     └────────────────────────────────────────────────────────────────────────────────────┘
                                      (闭环)
```

**关键能力：**
- **模型注册表**：版本管理、依赖追踪、模型血缘
- **AB 测试**：新旧模型并行运行，对比效果
- **灰度发布**：模型先在 10% 产线上运行，验证后全量推广
- **性能监控**：准确率、延迟、吞吐量实时监控

### 4.2 特征存储（Feature Store）

特征存储解决"特征复用"问题：

```python
# 特征存储示例：多个模型共享同一组特征
from feature_store import FeatureStore

fs = FeatureStore()

# 注册特征组
fs.create_feature_group(
    name='process_parameters',
    features=['temperature_mean', 'pressure_std', 'cycle_time_cv'],
    source='mes_process_data',
    update_frequency='realtime'
)

# 模型A（质量预测）和模型B（设备健康度）共用同一特征
features_for_quality = fs.get_features(
    group='process_parameters',
    entity='batch_001',
    as_of='2026-06-20T10:00:00'
)
```

### 4.3 边缘 AI vs 云端 AI

这是 AI Native 工厂架构设计中最关键的决策之一：

| 维度 | 边缘 AI | 云端 AI |
|------|--------|--------|
| 延迟 | <10ms | 100-500ms |
| 带宽 | 仅上传结果 | 需上传原始数据 |
| 模型大小 | 受限（量化/剪枝） | 不限 |
| 训练能力 | 弱（仅推理） | 强（GPU 集群） |
| 适用场景 | 实时质检、安全联锁 | 需求预测、全局优化 |

**推荐架构：边缘-云协同**

```
┌──────────────────┐          ┌──────────────────┐
│    边缘节点       │          │    云端平台       │
│  (产线侧)        │          │  (工厂级/集团级)  │
│                  │          │                  │
│  • 实时推理      │◄────────►│  • 模型训练      │
│  • 数据预处理    │  模型    │  • 特征工程      │
│  • 本地决策      │  下发    │  • 全局优化      │
│  • 异常缓存      │  数据    │  • 模型仓库      │
│                  │  上传    │  • 效果分析      │
└──────────────────┘          └──────────────────┘
```

**选型决策框架：**

1. **延迟 < 100ms？** → 边缘 AI
2. **需要 GPU 级算力？** → 云端 AI
3. **断网时必须运行？** → 边缘 AI
4. **需要跨产线数据？** → 云端 AI
5. **模型更新频率 > 每天？** → 边缘 AI（本地缓存 + 定期同步）

## 五、第三层：智能应用

智能应用层是 AI 能力的具体业务化封装。

### 核心应用矩阵

| 应用领域 | 典型场景 | 输入数据 | AI 能力 | 输出 |
|---------|---------|---------|---------|------|
| 质量管控 | 不良预测 | 工艺参数 | 分类模型 | 预警 + 参数建议 |
| 设备运维 | 预测性维护 | 振动/温度时序 | 异常检测 | 健康度评分 + 维护建议 |
| 生产排程 | 智能排产 | 订单/产能/物料 | 优化算法 | 最优排产方案 |
| 物流仓储 | AGV 调度 | 实时位置/任务 | 路径优化 | 调度指令 |
| 能源管理 | 能耗优化 | 用能曲线/产排 | 时序预测 | 用能策略 |

### 应用开发原则

1. **MLOps 原生**：每个应用从设计之初就考虑模型的持续迭代
2. **人机协同**：AI 给建议，人做决策，系统记录决策以反哺模型
3. **可解释性**：每个 AI 决策都附带理由和置信度
4. **可降级**：AI 故障时自动降级为规则引擎，保证生产安全

## 六、第四层：运营中台

运营中台将分散的智能应用整合为一体化的运营管理体系。

### 核心模块

**1. 全局优化引擎**

不是单个工序的最优，而是全厂全局最优。例如：质量预测模型建议降低温度，但能耗模型显示升温更节能，全局优化引擎需要在质量和能耗之间找到帕累托最优。

**2. 数字孪生**

构建工厂的数字镜像，在虚拟环境中模拟不同决策的效果：

```
物理工厂                        数字孪生
┌──────────────┐              ┌──────────────┐
│  实际生产     │──实时数据───▶│  虚拟映射     │
│              │              │              │
│  执行决策     │◀──优化方案───│  模拟推演     │
└──────────────┘              └──────────────┘
```

**3. 知识管理**

将工程师的经验、历史决策、改善案例结构化为知识图谱，与 AI 模型互补。

## 七、实施路径：从 0 到 1 构建 AI Native 工厂

### 分阶段实施路线图

**阶段一：数据基建期（0-6个月）**
- 完成关键工序的 IoT 部署
- 建设统一数据湖
- 实现数据治理和质量监控
- **里程碑**：数据可用性评分 > 80%

**阶段二：单点突破期（6-12个月）**
- 选择 2-3 个高价值场景构建 AI 应用
- 建立 MLOps 基础设施
- 验证模型效果和业务价值
- **里程碑**：至少 1 个场景实现闭环运行

**阶段三：平台化期（12-18个月）**
- 建设 AI 平台，实现模型管理标准化
- 扩展到 5-10 个应用场景
- 建立边缘-云协同架构
- **里程碑**：AI 平台支持新模型 2 周内上线

**阶段四：全面智能化（18-24个月）**
- 构建运营中台，实现全局优化
- 部署数字孪生
- 建立持续优化的运营机制
- **里程碑**：AI 决策覆盖 > 60% 运营场景

## 自查表：你的工厂离 AI Native 还有多远？

| 评估维度 | Level 1 | Level 2 | Level 3 | Level 4 (AI Native) |
|---------|---------|---------|---------|---------------------|
| 数据采集 | 关键工序有传感器 | 全工序覆盖 | 统一数据湖 | 实时数据管道 |
| 数据治理 | 无标准 | 有基本规范 | 元数据管理 | 自动化质量监控 |
| AI 应用 | 无 | 1-2个外挂模型 | 5+个应用 | 平台化运营 |
| 模型管理 | 手工开发 | 有版本管理 | CI/CD | 自动重训练 |
| 决策模式 | 人工决策 | AI辅助 | AI建议人审 | AI决策人监督 |
| 组织能力 | 无AI团队 | 有数据分析师 | 有ML工程师 | MLOps团队 |

> 大多数工厂目前处于 Level 1-2 之间。目标是在 12-18 个月内达到 Level 3，24 个月内向 Level 4 迈进。

## 结语

AI Native 工厂不是一场技术采购，而是一次组织变革。它要求工厂从"系统驱动"转向"数据驱动"，从"经验决策"转向"智能决策"，从"项目制AI"转向"平台化AI"。

**最关键的不是技术，而是思维方式的转变。** 当你开始用"数据能告诉我什么"替代"我觉得应该怎么做"时，你就已经踏上了 AI Native 之路。
